4. Flujo de Datos Detallado

sequenceDiagram participant SC as SwellFLClientMQTT participant FB as fog.py
Broker participant BC as FogClientSwell
Bridge participant SS as MQTTFedAvgSwell
Server SC->>SC: train_local() SC->>SC: weights = model(...) SC->>FB: publish(fl/updates)
{weights, samples} FB->>FB: buffer += weights FB->>FB: if len(buffer) >= K FB->>FB: partial = weighted_avg(buffer) FB->>BC: publish(fl/partial)
{partial_weights} BC->>BC: on_message() recibe partial BC->>BC: self.partial_weights = data BC->>SS: fit(parameters, config) SS->>SS: aggregate_fit([partial1, partial2]) SS->>SS: global = fedavg(partials) SS->>SC: publish(fl/global_model)
{global_weights} SC->>SC: on_message() recibe global SC->>SC: self.global_model = data SC->>SC: Próxima ronda: train_local()

📝 Notas Importantes

  • Herencia Múltiple: FogClientSwell/Sweet heredan de BaseMQTTComponent Y fl.client.NumPyClient
  • Aggregación en Fog: fog.py usa solo NumPy (no Flower) para agregación regional
  • MQTT + gRPC: Comunicación en dos niveles: MQTT (edge-fog) + gRPC (fog-central)
  • Publishing de Servidor: Servidores Flower publican global model en MQTT para que clientes lo reciban
  • Pattern Strategy: MQTTFedAvgSwell/Sweet extienden FedAvg de Flower
